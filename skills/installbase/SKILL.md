# /installbase — 플랫폼 전략 평가

## 무엇인가

기술 결정, 제품 전략, API 설계를 평가할 때 **설치 기반(install base)** 관점에서 다시 생각하는 프레임워크.

핵심 질문: *"이것이 개발자/사용자/생태계 풀을 넓히는가, 아니면 좁히는가?"*

---

## 트리거

Invoke when:
- `/installbase` — 기술 스택 선택, API 설계, 파트너십 결정
- "오픈소스 vs. 독점" 논쟁이 있을 때
- 개발자 도구, 플랫폼, 생태계 관련 결정
- VC 프레젠테이션에서 "경쟁 우위"를 논할 때

---

## 입력

- **제안된 결정** — 기술 선택, API 공개 여부, 플랫폼 접근 방식
- **시차** — 단기(분기) vs. 장기(3년+) 영향

---

## 워크플로우

### 1. 직접 의존자 명시

- 누구의 행동이 이 결정에 의존하는가?
- 그들이 더 많아지는가, 더 적어지는가?
  *(예: CUDA 사용자 수, API를 интегрирующий 서비스 수)*

### 2. Install Base 경로 비교

| | CLOSED 경로 | OPEN 경로 |
|---|---|---|
| 단기 | 수익 가능 | 수익 희생 |
| 사용자 | 고객 잠금 | 빠른 사용자 획득 |
| 생태계 | 느린 구축 | 생태계가 재현 |
| 플랫폼 가치 | 낮음 | 사용자² — 복합 효과 |

### 3. 시차 테스트

- "3년 후에 이 생태계가 없다면 무엇이 잘못되는가?"
- "3년 후에 100만 개발자가 있다면 무엇이 가능한가?"

### 4. 결론: OPEN / CLOSED / 조건부

## Plattform 가치 공식

```
V = U² × E

V = 플랫폼 가치
U = 사용자/개발자 수
E = 생태계 활성화도

→ U가 2배면 V는 4배
→ 사용자 획득은 단기 수익보다 중요할 수 있다
```

## kapan使用时 (When Not to Use)

- **생존이 먼저인 초기:** 수익 없이는 install base를 만들 시간이 없다
- **순환 시간이 빠른 소비자 앱:** 사용자 수보다 전환 비용이 중요
- **법적 규제 영역:** 보안/개인정보 때문에 개방이 불가능한 경우
